Cloud Native Concepts

    Infrastructure As Code
    • Use code to provision your infra, not point & click interface or tickets
    • Code, infra, config versionned together
    • Consistency between dev / test / prod / DR
    • Implemented mostly through TerraForm. Ansible / Chef as additional tools.

Cloud Native Concepts

    CICD
    • Automate build, linting, packaging, testing
    • Detect bugs and integration issues as early as possible
    • CD is very close to CI technically speaking, it is more an org change. Can go all the way to canary testing and baking period.

Cloud Native Concepts

    Microservices
    • On the surface, just another iteration for SOA with a lighter approach - eg. REST vs. SOAP
    • "Pets vs. Cattle" analogy: any server can die and be replaced at any time
    • Relies on open standards, try to be as close to the web as possible, but sometimes people backtrack (eg. DNS vs. Open Service Broker)
    • 'micro' is a relative thing. For some companies, 'micro' might mean one service for a 20-people team or 5 services for a 5-people Scrum team.

Cloud Native Concepts

    DevOps
    • "Dev doing their own Ops"
    • World Before: Dev files tickets for Ops, and they are the gate keepers. Lot of friction & debate to decide what changes are safe, and Ops can quickly become overloaded by change requests.
    • World After: Tools are created to enable dev to release, monitor and scale their app
    • Do you need a DevOps team ?
        Bad Pattern: create a new DevOps team, that are just another layer in the org between Dev and Ops.
        Good Pattern: the DevOps team is small, acts as a mentor in the org to foster good practices and create tools to automate processes.
